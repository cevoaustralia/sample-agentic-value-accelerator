# seeding_scripts

Database seeding pipeline for the Market Surveillance system. Three-module architecture:

1. **`data_gen`** -- Generate synthetic CSV data (no database required)
2. **`database_seeding`** -- Load CSV files into PostgreSQL
3. **`db_ops`** -- Database lifecycle management (create schema, clean, verify, export)

---

## Directory Structure

```
seeding_scripts/
├── requirements.txt                # Python dependencies (psycopg, PyYAML)
├── sample_conversations.txt        # Reference text for synthetic eComm generation
│
├── data_gen/                       # Module 1: Synthetic data generation
│   ├── generate.py                 # CLI orchestrator (entry point)
│   ├── models.py                   # Dataclass definitions for all record types
│   ├── constants.py                # Hardcoded reference data and lookup values
│   ├── csv_writer.py               # Utility: write dataclass lists to CSV
│   ├── gen_references.py           # Generates 29 ref_* table CSVs
│   ├── gen_dimensions.py           # Generates dim_account, dim_product, dim_actor
│   ├── gen_trades.py               # Generates fact_trade
│   ├── gen_flagged_trade.py        # Generates flagged_trade (alert-to-trade bridge)
│   ├── gen_alerts.py               # Generates fact_alert
│   ├── gen_ecomm.py                # Generates fact_ecomm (IM/RFQ messages)
│   └── gen_ecomm_synth.py          # Generates synthetic eComm from sample_conversations.txt
│
├── database_seeding/               # Module 2: CSV-to-PostgreSQL loader
│   ├── seed.py                     # CLI orchestrator with CSV->table mappings
│   ├── csv_loader.py               # Reads CSV via DictReader, applies type coercion
│   ├── table_inserter.py           # Builds dynamic INSERT queries, batches of 1000
│   └── db_connection.py            # PostgreSQL connection manager with retry logic
│
├── db_ops/                         # Module 3: Database lifecycle management
│   ├── db_init.py                  # Create tables from schema YAML (DDL generation)
│   ├── db_clean.py                 # Truncate or drop all tables
│   ├── db_verify.py                # Inspect schema, show table structure and counts
│   ├── db_dump.py                  # Export data to CSV/console with validation
│   └── schema_config_yaml_*.yaml   # Schema definition (source of truth for DDL)
│
├── data/                           # Pre-existing source CSVs (customer data + eComm synthetic data)
│
└── synthetic_data/                 # Generated CSV output (consumed by database_seeding)
```

## Prerequisites

Create a virtual environment and install dependencies:

```bash
cd seeding_scripts
python -m venv seeding_env
source seeding_env/bin/activate
pip install -r requirements.txt
```

This installs:
- `psycopg[binary]` -- PostgreSQL adapter (required for `database_seeding` and `db_ops`)
- `PyYAML` -- YAML parser (required for `db_ops`)

The `data_gen` module has **no external dependencies** (stdlib only).

> **Note:** All `python -m seeding_scripts.*` commands must be run from the project root directory (the parent of `seeding_scripts/`).

## Pre-existing Data

The repo includes pre-existing data files in **`data/`** that are ready to use without running `data_gen`:

- **`data/`** -- Source CSVs with customer-provided data (e.g. `fact_trade.csv`, `dim_account.csv`) and pre-existing eComm synthetic data (`fact_ecomm_synth.csv`). This is the primary location for all pre-existing data.
- **`synthetic_data/`** -- Output directory populated by `data_gen`. When you run the generator, it copies customer CSVs from `data/` and writes newly generated CSVs here. The `database_seeding` module reads from this directory.

> **eComm data note:** The eComm synthetic data file (`fact_ecomm_synth.csv`) is **not** generated by the `data_gen` module. Pre-existing synthetic eComm data is provided in the `data/` directory. When you run `data_gen`, this file is copied to `synthetic_data/` alongside the generated files, and the `database_seeding` module then loads it into the `fact_ecomm` table.

## Module 1: `data_gen` -- Generate Synthetic Data

Produces CSV files in `synthetic_data/` with no database connection required. Uses reproducible random generation (via `--seed`) across 7 phases that run in dependency order.

### Generation Phases

| Phase | Output | Depends On |
|-------|--------|------------|
| 1. `references` | 29 `ref_*.csv` files | None |
| 2. `dimensions` | `dim_account.csv`, `dim_product.csv`, `dim_actor.csv` | references |
| 3. `trades` | `fact_trade.csv` | dimensions |
| 4. `flagged_trade` | `flagged_trade.csv` | trades |
| 5. `alerts` | `fact_alert.csv` | dimensions, flagged_trade |
| 6. `ecomm` | `fact_ecomm.csv` | dimensions |
| 7. `ecomm_synth` | `fact_ecomm_synth.csv` | dimensions, `sample_conversations.txt` |

### Usage

```bash
# Generate all CSVs with defaults 
python -m seeding_scripts.data_gen \
    --output-dir ./seeding_scripts/synthetic_data

# Generate with custom counts and a fixed seed for reproducibility
python -m seeding_scripts.data_gen \
    --seed 42 \
    --output-dir ./seeding_scripts/synthetic_data \
    --accounts 10 --products 15 --actors 20 \
    --trades 50 --alerts 10 --ecomm 20
```

> **Tip:** To generate completely new/different synthetic data, use `--seed <new_number>` with a different seed value. Each seed produces a deterministic but distinct dataset.

### CLI Flags

| Flag | Default | Description |
|------|---------|-------------|
| `--seed` | None | Random seed for reproducible output |
| `--output-dir` | `./seeding_scripts/synthetic_data` | Output directory for CSV files |
| `--data-dir` | `./seeding_scripts/data` | Directory with pre-existing customer CSV files |
| `--accounts` | 10 | Number of accounts to generate |
| `--products` | 15 | Number of products to generate |
| `--actors` | 20 | Number of actors to generate |
| `--trades` | 50 | Number of trades to generate |
| `--alerts` | 10 | Number of alerts to generate |
| `--ecomm` | 20 | Number of eComm conversations to generate |
| `--start` | None | Start from this phase |
| `--end` | None | Stop after this phase |

## Module 2: `database_seeding` -- Load CSVs into PostgreSQL

Reads CSV files from a directory and inserts them into PostgreSQL tables in foreign-key dependency order. This module is decoupled from `data_gen` -- it works with any CSV files that match the expected filenames and column headers.

### Usage

```bash
# Load all CSVs into the database
python -m seeding_scripts.database_seeding \
    --input-dir ./seeding_scripts/synthetic_data \
    --database <database_name> --host localhost --port 5432 --user <username>

# Clear tables before loading (delete existing data first)
python -m seeding_scripts.database_seeding \
    --input-dir ./seeding_scripts/synthetic_data \
    --database <database_name> --clear
```

### CLI Flags

| Flag | Default | Description |
|------|---------|-------------|
| `--input-dir` | (required) | Directory containing CSV files |
| `--database` | `$DB_NAME` | PostgreSQL database name |
| `--host` | `$DB_HOST` or `localhost` | Database host |
| `--port` | `$DB_PORT` or `5432` | Database port |
| `--user` | `$DB_USER` or `postgres` | Database user |
| `--password` | `$DB_PASSWORD` | Database password |
| `--clear` | False | Delete existing rows before loading |
| `--tables` | None | Load only specific tables (by DB table name) |

### Set the Database Credentials

Retrieve the credentials from AWS Secrets Manager:

```bash
export DB_SECRET=$(aws secretsmanager get-secret-value --secret-id market-surveillance-db-dev --query SecretString --output text --region us-east-1)
export DB_PASSWORD=$(echo $DB_SECRET | jq -r '.PASSWORD')
export DB_NAME=$(echo $DB_SECRET | jq -r '.DBNAME')
export DB_USERNAME=$(echo $DB_SECRET | jq -r '.USERNAME')
```

These environment variables are used by the CLI flags (`--database`, `--user`, `--password`) and can also be referenced directly.

## Module 3: `db_ops` -- Database Lifecycle Management

Four utilities for managing the PostgreSQL database schema and data.

### `db_init` -- Create Tables from Schema YAML

Reads the schema YAML file and generates DDL (CREATE TABLE, ALTER TABLE ADD CONSTRAINT) in an 8-level dependency order.

```bash
# Create all tables
python -m seeding_scripts.db_ops.db_init \
    --database mydb --host localhost --port 5432 --user postgres

# Drop and recreate all tables
python -m seeding_scripts.db_ops.db_init \
    --database mydb --recreate
```

> **`--recreate` flag:** This drops all foreign keys and tables, then recreates them from the schema YAML. Use this when you need to reset the database schema (e.g., after schema changes). This is a destructive operation -- all existing data will be lost.

### `db_clean` -- Truncate or Drop Tables

```bash
# Truncate all tables (preserve schema, delete data)
python -m seeding_scripts.db_ops.db_clean \
    --database mydb

# Drop all tables entirely
python -m seeding_scripts.db_ops.db_clean \
    --database mydb --drop
```

### `db_verify` -- Inspect Schema and Data

```bash
# Show all tables with row counts
python -m seeding_scripts.db_ops.db_verify \
    --database mydb --counts

# Show details for a specific table
python -m seeding_scripts.db_ops.db_verify \
    --database mydb --table dim_account
```

### `db_dump` -- Export and Validate Data

```bash
# Export all tables to CSV
python -m seeding_scripts.db_ops.db_dump \
    --database mydb --format csv --output ./db_export

# Export specific tables
python -m seeding_scripts.db_ops.db_dump \
    --database mydb --table dim_account fact_trade
```

## End-to-End Workflow

The typical workflow to set up the database from scratch using the pre-existing data in `data/`:

```bash
# Step 1: Open the Aurora tunnel (keep this terminal open)
make port-forward

# Step 2 (separate terminal): Set the database credentials
export DB_SECRET=$(aws secretsmanager get-secret-value --secret-id market-surveillance-db-dev --query SecretString --output text --region us-east-1)
export DB_PASSWORD=$(echo $DB_SECRET | jq -r '.PASSWORD')
export DB_NAME=$(echo $DB_SECRET | jq -r '.DBNAME')
export DB_USERNAME=$(echo $DB_SECRET | jq -r '.USERNAME')

# Step 3: Create database schema (tables + foreign keys)
python -m seeding_scripts.db_ops.db_init \
    --database $DB_NAME --host localhost --port 5940 --user $DB_USERNAME

# Step 4: Load pre-existing CSVs into the database
python -m seeding_scripts.database_seeding \
    --input-dir ./seeding_scripts/data \
    --database $DB_NAME --host localhost --port 5940 --user $DB_USERNAME

# Step 5: Verify the loaded data
python -m seeding_scripts.db_ops.db_verify \
    --database $DB_NAME --host localhost --port 5940 --user $DB_USERNAME --counts
```

> **Note:** If you want to generate new synthetic data instead of using the pre-existing CSVs, use the `data_gen` module first (see [Module 1](#module-1-data_gen----generate-synthetic-data)), then point `--input-dir` to `./seeding_scripts/synthetic_data`.

To reset and re-seed:

```bash
python -m seeding_scripts.db_ops.db_init \
    --database $DB_NAME --host localhost --port 5940 --user $DB_USERNAME --recreate
python -m seeding_scripts.database_seeding \
    --input-dir ./seeding_scripts/data \
    --database $DB_NAME --host localhost --port 5940 --user $DB_USERNAME
```

### Connecting via Port Forwarding (Aurora)

The Aurora PostgreSQL database runs inside a private VPC subnet and is not directly accessible from your local machine. To connect, the project uses an EC2 Instance Connect (EIC) bastion host that tunnels traffic from a local port to the Aurora endpoint over SSH.

#### How It Works

```
Local machine ──(localhost:5940)──> Bastion Host ──(private subnet)──> Aurora PostgreSQL :5432
                   SSH tunnel via           EC2 Instance Connect
                   aws ec2-instance-connect
```

1. `aurora-port-forward.sh` resolves the Aurora cluster endpoint (from Terraform output or by querying RDS)
2. It locates the running bastion host EC2 instance (`market-surveillance-dev-bastion-host`)
3. It establishes an SSH port-forward using `aws ec2-instance-connect ssh --local-forwarding`

#### Quick Start

From the project root:

```bash
make port-forward
```

This opens a tunnel from **`localhost:5940`** to Aurora port `5432`. Keep this terminal session open -- the tunnel stays active as long as the SSH connection is alive. Then follow the steps in the [End-to-End Workflow](#end-to-end-workflow) section in a separate terminal.

#### Configuration

The port-forwarding script reads these environment variables (all have defaults):

| Variable | Default | Description |
|----------|---------|-------------|
| `AWS_REGION` | `us-east-1` | AWS region where the infrastructure is deployed |
| `APP_NAME` | `market-surveillance` | Application name (used to find the bastion host and RDS cluster) |
| `ENV_NAME` | `dev` | Environment name (`dev`, `staging`, `prod`) |
| `LOCAL_PORT` | `5940` | Local port to bind the tunnel to |

#### Prerequisites

- **AWS CLI v2** installed and configured with credentials that can access the target account
- **EC2 Instance Connect CLI** (`aws ec2-instance-connect ssh` support)
- A running bastion host EC2 instance tagged `{APP_NAME}-{ENV_NAME}-bastion-host`
- Network connectivity from the bastion host to the Aurora cluster (same VPC / security group rules)

#### Troubleshooting

| Problem | Fix |
|---------|-----|
| `Could not find RDS endpoint` | Verify the Aurora cluster exists: `aws rds describe-db-clusters --region us-east-1` |
| `No running instance found` | Check the bastion host is running: `aws ec2 describe-instances --filters "Name=tag:Name,Values=market-surveillance-dev-bastion-host"` |
| `Connection refused on localhost:5940` | The tunnel session may have dropped -- re-run `make port-forward` |
| `psycopg connection timeout` | Ensure you're using port `5940` (not `5432`) in your `--port` flag |
| `Permission denied` | Verify your AWS credentials have `ec2-instance-connect:SendSSHPublicKey` and `ec2:DescribeInstances` permissions |
| `Table already exists / relation conflict` | Run `python -m seeding_scripts.db_ops.db_init --database mydb --recreate` to drop and recreate all tables from the schema YAML. This resolves conflicts from stale or mismatched table definitions. |
